GLOBAL TECH UNIVERSITY — COURSE SYLLABUS
CS280: SOFTWARE ENGINEERING II
Credits: 3 | Lecture: 3 hours/week
Department: Computer Science and Engineering
Prerequisites: CS160 (minimum grade C) AND CS201 (minimum grade C)

COURSE DESCRIPTION


CS280: Software Engineering II is the advanced continuation of CS160, taking students from the principles of software engineering introduced in the first course into the practices, architectures, and engineering culture of large-scale, production-grade software development. Where CS160 introduced the software development lifecycle, Agile methodology, and foundational design patterns through a semester-long team project, CS280 operates at a higher level of technical and organizational complexity — examining how the principles of software engineering scale from a team of five students building a course project to an organization of five hundred engineers building a distributed, continuously deployed, globally available software product.

The course is organized around four major themes. The first is software architecture at scale: students study architectural styles beyond those introduced in CS160, including event-driven architectures, CQRS (Command Query Responsibility Segregation), event sourcing, and serverless architectures, and apply these patterns to the design of distributed systems. The second is quality engineering: students study advanced testing strategies — property-based testing, mutation testing, chaos engineering, and performance testing — and develop a sophisticated understanding of software quality beyond functional correctness. The third is DevOps and continuous delivery: students implement complete CI/CD pipelines, study infrastructure as code (IaC), container orchestration (Kubernetes), and the SRE (Site Reliability Engineering) philosophy. The fourth is software engineering in practice: students examine real-world case studies of software engineering successes and failures, study the economics of technical debt, and develop the leadership and communication skills needed to function effectively in senior technical roles.

Students complete a semester-long capstone engineering project — larger and more architecturally complex than the CS160 team project — that requires them to design, build, deploy, and operate a distributed software system with automated testing, CI/CD, and monitoring in place.

LEARNING OBJECTIVES

Upon successful completion of CS280, students will be able to:

Design and evaluate software architectures for large-scale distributed systems, applying appropriate architectural styles (microservices, event-driven, CQRS, serverless).
Apply the twelve-factor app methodology to the design of cloud-native applications.
Implement and reason about event-driven architectures using message queues and event streams (Apache Kafka at a conceptual level).
Design and implement RESTful APIs following industry best practices (OpenAPI specification, versioning, error handling, pagination).
Apply advanced software design patterns: saga pattern for distributed transactions, circuit breaker, bulkhead, retry with exponential backoff.
Design and implement property-based tests using hypothesis testing frameworks and explain their advantages over example-based tests.
Apply mutation testing to evaluate the quality of a test suite and interpret mutation scores.
Design and execute performance tests (load testing, stress testing, soak testing) using tools such as k6 or Locust.
Implement a complete CI/CD pipeline using GitHub Actions or GitLab CI, including automated testing, static analysis, containerization, and deployment.
Write infrastructure as code using Terraform or Ansible and explain the benefits of IaC over manual configuration.
Deploy and manage containerized applications using Docker and Kubernetes (at a practical level).
Implement distributed system observability: structured logging, metrics collection (Prometheus), distributed tracing (OpenTelemetry).
Define and measure Service Level Objectives (SLOs) and Service Level Indicators (SLIs), and apply error budget concepts.
Quantify and manage technical debt using measurement techniques and make principled trade-off decisions about debt repayment.
Conduct a post-mortem analysis of a software incident, applying blameless retrospective principles.
Lead technical discussions, write architectural decision records (ADRs), and communicate engineering decisions to non-technical stakeholders.
Analyze historical software engineering failures (Therac-25, Knight Capital, Boeing 737 MAX MCAS) and extract engineering and organizational lessons.

WEEKLY COURSE BREAKDOWN

Week 1: From CS160 to CS280 — Software Engineering at Scale
Topics: Review of CS160 concepts, what changes at scale, Conway's Law, organizational structures and their impact on software architecture, the microservices vs. monolith debate, course project introduction.
Assignments: Team formation; architectural critique of a real-world system (written).

Week 2: Advanced Software Architecture — Distributed Systems Patterns
Topics: The twelve-factor app, microservices deep dive, service mesh (Istio overview), event-driven architecture, message queues (RabbitMQ, Kafka), publish-subscribe vs. request-response.
Assignments: Problem set 1 — architectural pattern selection and justification; Architecture diagram of assigned system.

Week 3: API Design — REST, GraphQL, and gRPC
Topics: RESTful API design principles, OpenAPI specification, versioning strategies, GraphQL vs. REST trade-offs, gRPC and Protocol Buffers, API security (OAuth 2.0, JWT).
Assignments: Problem set 2 — API design exercise; OpenAPI specification for team project.

Week 4: Distributed System Challenges — Consistency, Availability, Partitioning
Topics: The CAP theorem revisited, consistency models (strong, eventual, causal), distributed transactions and the saga pattern, idempotency, the two-generals problem, the Byzantine generals problem.
Assignments: Problem set 3 — distributed systems design questions; Quiz 1.

Week 5: Advanced Design Patterns for Distributed Systems
Topics: Circuit breaker pattern, bulkhead pattern, retry with exponential backoff and jitter, CQRS and event sourcing, the outbox pattern for reliable event publishing.
Assignments: Problem set 4 — pattern application exercise; Team project milestone 1 (architecture design document + ADRs).

Week 6: Quality Engineering I — Advanced Testing Strategies
Topics: Limitations of unit and integration testing, property-based testing (hypothesis framework), mutation testing (mutmut/PIT), contract testing (Pact), the testing trophy vs. testing pyramid debate.
Assignments: Property-based testing lab; Mutation score improvement exercise.

Week 7: Quality Engineering II — Performance and Chaos Engineering
Topics: Performance testing types (load, stress, soak, spike), k6/Locust for load generation, performance metrics (latency percentiles, throughput, error rate), chaos engineering principles (Netflix Chaos Monkey), GameDay exercises.
Assignments: Performance test implementation for team project; Chaos experiment design document.

Week 8: Midterm Examination and Architecture Review
Topics: Comprehensive review of Weeks 1–7; Midterm Exam; Team project architecture review presentation (peer + instructor evaluation; detailed feedback on design decisions).
Assignments: Midterm Exam; Architecture review presentation.

Week 9: CI/CD Pipelines — Design and Implementation
Topics: CI/CD philosophy, pipeline stages (build, test, static analysis, security scan, containerize, deploy), GitHub Actions workflow design, branch protection rules, feature flags, canary deployments, blue-green deployments.
Assignments: Implement complete CI/CD pipeline for team project (graded lab).

Week 10: Infrastructure as Code and Container Orchestration
Topics: The case for IaC, Terraform fundamentals (providers, resources, state), Kubernetes architecture (pods, deployments, services, ingress, ConfigMaps, Secrets), Helm charts, the GitOps model.
Assignments: Terraform lab — provision cloud infrastructure; Kubernetes deployment of team project.

Week 11: Observability — Logging, Metrics, and Tracing
Topics: The three pillars of observability, structured logging (JSON logs, log aggregation), metrics (Prometheus data model, PromQL, Grafana dashboards), distributed tracing (OpenTelemetry, trace propagation), alerting design.
Assignments: Implement observability stack for team project; Dashboard design exercise.

Week 12: Site Reliability Engineering — SLOs, Error Budgets, and Incident Management
Topics: The SRE philosophy (from Google SRE book), SLIs and SLOs, error budgets and their use in prioritization, on-call practices, incident management (detection, response, escalation, communication), blameless post-mortems.
Assignments: Define SLOs for team project; Write a synthetic post-mortem for a provided incident scenario.

Week 13: Technical Debt — Measurement, Management, and Trade-offs
Topics: Technical debt taxonomy (deliberate vs. inadvertent, reckless vs. prudent), measuring technical debt (static analysis tools, code churn metrics, cyclomatic complexity), the debt quadrant, making principled trade-off decisions, refactoring strategies for legacy systems.
Assignments: Technical debt audit of team project codebase; Debt reduction plan document.

Week 14: Software Engineering Ethics and Historical Failures
Topics: Engineering ethics frameworks, the Therac-25 case study (software-caused radiation overdoses), the Knight Capital Group incident ($440M loss in 45 minutes), the Boeing 737 MAX MCAS system (737 deaths), lessons for engineering culture and process.
Assignments: Case study analysis essay (individual, 1,500 words); Final project work session.

Week 15: Final Project Presentations and Professional Retrospective
Topics: Team project final presentations (distributed system live demo, architecture walkthrough, observability demo, CI/CD pipeline demo), professional retrospective, career pathways in software engineering.
Assignments: Final Project complete system delivery; Presentation; Individual reflection essay.

ACADEMIC CONTEXT

CS280 exists because CS160, however excellent, cannot fully prepare students for the realities of engineering software at the scale and complexity of modern production systems. The gap between a well-designed semester project and a production-grade distributed system is enormous, and it is a gap that has historically caught new graduate engineers off guard. CS280 is designed to close that gap as far as a university course can, by immersing students in the tools, practices, and thinking of senior software engineers and site reliability engineers working on real-world systems.

The prerequisite of CS201 alongside CS160 is important because the distributed systems content of CS280 draws heavily on algorithmic reasoning. The CAP theorem, consensus algorithms, and distributed transaction patterns require the kind of formal reasoning about correctness and failure modes that CS201 develops. A student who has only CS160 — who understands software engineering processes but lacks the algorithmic rigor of CS201 — is not equipped to reason about why eventual consistency is a meaningful trade-off, or why the two-generals problem is not merely a curiosity but a fundamental impossibility result that constrains the design of every distributed system ever built.

The case studies in Week 14 — Therac-25, Knight Capital, and the Boeing 737 MAX — are not merely cautionary tales but primary sources for understanding how software engineering failures happen and how they can be prevented. The Therac-25 case, in which software race conditions in a radiation therapy machine caused six patients to receive massive radiation overdoses (three fatally), illustrates the lethal consequences of software defects in safety-critical systems and the dangers of removing hardware safety interlocks without adequate software validation. The Knight Capital incident, in which a deployment error activated untested legacy code and caused the firm to lose $440 million in 45 minutes, illustrates the catastrophic consequences of poor deployment practices and the absence of automated rollback mechanisms. The Boeing 737 MAX case, in which inadequate software requirements analysis and incomplete pilot training contributed to two crashes killing 346 people, illustrates the systemic failures of organizational culture, regulatory capture, and engineering accountability that can occur when commercial pressures override engineering judgment. Together, these cases make the abstract principles of software engineering — rigorous testing, safe deployment, observability, blameless post-mortems — concrete and morally urgent in a way that no amount of theoretical argument can achieve.

The introduction of CI/CD pipelines, Kubernetes, and infrastructure as code in CS280 reflects the transformation of software engineering practice over the past decade. The DevOps movement, which emerged from the recognition that the traditional separation between development and operations teams created costly friction and reliability problems, has fundamentally changed how software is built and deployed. Students who graduate from GTU with hands-on experience implementing CI/CD pipelines, deploying containerized applications to Kubernetes, and writing Terraform configurations are equipped with skills that are immediately applicable in internship and entry-level positions at companies ranging from early-stage startups to major technology enterprises. The investment GTU makes in providing access to cloud infrastructure for CS280 projects is among the most practically impactful investments in the curriculum.
